# 项目意图（Project）

<!-- 本文件是项目的"为什么"。人主笔、AI 只读。保持精简：每节几句话即可。 -->
一个面向旅行用户的 H5 行程查看应用。
产品仅负责“高效查看”，不承担“编辑/生成”职责。围绕日期、地点、内容、时长、准备事项、建议六类关键信息，帮助用户在最短时间内完成信息获取与出行准备。

## Vision｜愿景

<!-- 一句话说明：为谁、解决什么核心问题、达成什么长期价值。
示例：为中小商家提供"开箱即用"的在线收银，让非技术店主 5 分钟完成开店收款。 -->
为旅行用户提供一个“打开即看、快速理解”的轻量行程查看工具，在出行场景下保持清晰、低负担、高效率的体验。

## Goals｜目标

<!-- 列 3–5 条可衡量的近期目标（建议 OKR 风格：目标 + 关键结果）。
示例：
- 支持微信 / 支付宝 / 银行卡三种收款方式
- 单笔支付端到端 P95 < 2s -->
* 读取固定格式的行程信息并稳定展示，核心字段完整展示率 100%。
* 以 H5 形式展示行程数据，保证移动端信息层级清晰、重点突出。
* 将核心查看路径控制在 3 次交互内，用户可快速定位任意一天关键内容。
* 首屏加载性能目标：P95 < 1.5s（默认口径：4G + 主流中端机）。
* 在交互设计上坚持极简，不引入与“查看信息”无关的复杂流程。

## Non-Goals｜不做什么

<!-- 明确划出范围之外的事，防止范围蔓延（业界公认最有效的边界手段）。
示例：
- 不做线下 POS 硬件集成
- 不做跨境结算与多币种 -->
* 不生成行程信息，不修改行程信息。
* 不调用外部接口获取/同步/存储行程信息。
* 不实现账号体系、社交分享、评论、推荐等非核心能力。
* 不做复杂流程型交互（如多步骤向导、重配置化操作）。

## 信息优先级（展示顺序）

* 日期 > 地点 > 内容 > 时长 > 准备事项 > 建议。

## 成功标准｜Success Criteria

<!-- 怎样算成功？可量化的验收 / 北极星指标。
示例：上线 3 个月内 1000 家活跃商户，支付成功率 > 99.5%。 -->

* 信息清晰：用户可快速识别日期、地点、内容、时长、准备事项、建议六类信息。
* 操作简单：主要任务（查看当日与后续行程）可直达，学习成本低。
* 使用流畅：页面滚动与切换稳定，无明显卡顿或阻塞。
* 性能达标：首屏加载时间满足 P95 < 1.5s（4G + 主流中端机口径）。
* 范围可控：功能持续聚焦“查看”，不蔓延到“编辑/生成/同步”。
