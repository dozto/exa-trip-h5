# Skill: Feature Delivery（流程编排）

> 用途：把 feature 交付流程标准化执行，避免“直接开写”导致的边界漂移。
> 何时用：用户提出“新增/修改 feature”时立即启用。
> 触发：`/feature <name> [mode=fast|strict]`（默认 `fast`）。

## 执行协议

### Mode: fast | strict

- `fast`（默认）：最小提问、快速推进、低确认负担。
- `strict`：更严格澄清与风险核对，适合高风险变更。

模式差异：

- `fast`：每轮最多 1-2 个阻塞问题；待确认决策 <= 2。
- `strict`：每轮最多 3 个阻塞问题；待确认决策 <= 3，必须给出额外风险说明。

### Phase 1: 上下文加载

- 读取 `.human/*`、`.ai/index.md`、`.ai/current.md`。
- 输出 3-5 条“当前理解”。

### Phase 2: 澄清提问

- 默认采用“AI 主动驱动 + 最小提问”。
- 每轮最多提阻塞问题（fast: 1-2, strict: 1-3），其余使用默认假设推进。
- 每个问题必须给出推荐默认值与影响说明。

### Phase 2.1: 阻塞问题判定（先判断，再决定是否提问）

仅当满足任一条件时提问：

- 会改变领域边界/端口设计/实施顺序
- 涉及安全、合规、计费、生产不可逆操作
- 缺少验收标准导致无法判定完成

不满足以上条件：AI 直接采用默认方案推进，并记录到 RFC。

### Phase 2.2: 最小问题集（默认只问这 3 类）

1) 完成标准

- 问题：这次做到什么程度算完成？
- 推荐默认值：可用 MVP + 可验证测试通过。

2) 范围边界

- 问题：本次明确不做哪些内容？
- 推荐默认值：只交付单场景主路径，非关键扩展延后。

3) 硬约束

- 问题：是否有安全/合规/性能/预算硬约束？
- 推荐默认值：优先安全与稳定，性能按当前基线不退化。

### Phase 2.3: 扩展问题库（按需，不默认全问）

> 仅在 Phase 2.1 判定为“阻塞”时，从下列类别选择性追问，不允许全量罗列。

1) 业务目标与成功标准

- 这个 feature 最终要解决的核心问题是什么？
- 成功标准是什么（可量化）？
- 推荐默认值：若未给指标，先用“可用性优先 + 可观测性可验证”的验收标准。

2) 范围与非目标

- 本次必须交付哪些最小能力（MVP）？
- 明确不做哪些内容（非目标）？
- 推荐默认值：先做单场景 MVP，明确列出 Out of Scope。

3) 参与者与边界

- 谁是调用方（用户/系统）？谁是被影响方？
- 输入输出边界是什么（入口、返回、副作用）？
- 推荐默认值：只开放一个主入口（inbound port），其余走后续迭代。

4) 领域与一致性

- 涉及哪些 domain / bounded context？
- 核心聚合与不变量是什么？
- 是否跨聚合/跨上下文一致性？如何保证（事件/补偿）？
- 推荐默认值：单聚合事务优先，跨边界用 feature 显式编排与补偿实现最终一致。

5) 六边形拆分

- 需要哪些 inbound ports？
- 依赖哪些 outbound ports（DB/支付/消息/外部 API）？
- 哪些 inbound/outbound 实现需要新增或扩展？
- 推荐默认值：先定义 ports，再决定 adapter 实现细节。

6) 规则与术语

- 是否引入新业务术语？是否需要更新 `.human/glossary.md`？
- 哪些规则必须在 `src/domains` 保证（而非 feature/adapter）？
- 推荐默认值：新增术语一律登记 glossary 后再进入命名。

7) 质量属性与约束

- 性能、可靠性、安全、合规有哪些硬约束？
- 失败语义与重试/幂等策略是什么？
- 推荐默认值：明确失败可观测、幂等可重复执行、关键外部调用可重试。

8) 交付策略

- 是否分阶段上线？阶段划分如何做？
- 风险最高的环节是什么？
- 推荐默认值：先交付可验证最小闭环，再增量扩展。

### Phase 2.4: 首轮提问输出格式

- AI 先输出：`当前理解(3-5条)`
- 再输出：`问题清单(按模式限制数量)`
- 每个问题附：`推荐默认值` + `若不确认将导致的影响`
- 最后输出：`待你拍板的必须决策项`

### Phase 3: 拆分建议

- 给出 domain 候选、feature 切分、bounded context 边界、ports 设计建议。
- 标注“必须决策项”与“可延后项”。

### Phase 4: 文档产出（代码前）

- 生成/更新 RFC（状态 `pending`）。
- 需要时更新 `.ai/domains/*`、`.ai/features/*`、`.human/glossary.md`。
- 请求用户确认。

### Phase 5: 实现与验证（确认后）

- 按 TDD 执行：red -> green -> refactor。
- feature use case 统一使用函数/函数工厂实现；不使用 class + `new`。
- 合并前运行：`xp-validation` + `ddd-validation`。

### Phase 6: 回写收口

- 更新 `.ai/current.md`。
- RFC `pending -> accepted`。
- 汇报测试与校验证据。

## 失败即停止（硬约束）

- 未确认 RFC 即进入实现。
- 未写失败测试直接编码。
- 依赖方向违规仍继续推进。

## 对用户的最小输出模板

1. 需求理解摘要
2. 问题清单（含建议）
3. 领域/特性拆分方案
4. RFC 草案链接与待确认项
5. 实现后证据（测试/校验）
